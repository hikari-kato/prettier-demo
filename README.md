# Prettier 導入サンプル

Web ページのソースコード（HTML・CSS・JavaScript など）の**見た目の揃え方**を、ファイルを保存したときに自動で整えるための設定例です。

---

## 何のために使うか

同じ内容でも、人によって改行や余白の入れ方がバラバラになりがちです。  
Prettier を使うと、**保存するだけで**チーム内で揃った書き方になります。

- 手で体裁を直す手間が減る
- 「ここは改行を揃えて」といったやりとりが減る

---

## 導入手順（最低限）

### 1. 拡張機能を入れる

**VS Code** または **Cursor** で、次の拡張機能をインストールします。

- [Prettier - Code formatter](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode)

（拡張機能の検索欄に「Prettier」と入力すると見つかります）

### 2. 設定ファイルをプロジェクトに置く

プロジェクトの**一番上のフォルダ**に、次の 3 つを置きます。  
このリポジトリからそのままコピーして使えます。

| ファイル名              | ざっくりした役割               |
| ----------------------- | ------------------------------ |
| `.prettierrc`           | チームで揃えるルール           |
| `.prettierignore`       | 自動整形しないファイルの一覧   |
| `.vscode/settings.json` | 「保存したら整える」などの設定 |

### 3. プロジェクトを開いて試す

1. Cursor / VS Code で、このプロジェクトのフォルダを開く
2. `sample.html` を開く
3. わざと改行や余白を崩してから **保存**（Mac: `⌘ + S` / Windows: `Ctrl + S`）
4. 保存した瞬間に、きれいに揃えば成功です

同じ要領で、次のファイルでも試せます。

- `common/js/sample.js`
- `common/sass/sample.scss`

**PHP について:** Prettier は PHP ファイルには対応していません。`.php` を保存しても自動では整いませんが、不具合ではなく想定どおりの動きです。

## 新規プロジェクトと既存プロジェクト、どちらから始める？

### まず試すなら新規・小さなプロジェクトがおすすめ

このリポジトリのような**新規または小さなプロジェクト**で試すと、次のメリットがあります。

- 変更ファイルが少なく、Prettier の効果だけを確認できる
- 「保存したら揃う」体験に集中できる
- 機能の変更と混ざらず、説明やレビューがしやすい

チームへの説明や、個人での動作確認には、このやり方が向いています。

### 既存プロジェクトでも導入できる（差分は一時的に大きくなる）

本番の既存プロジェクトにも Prettier は導入できます。  
初回に一括で整形すると、**Git の差分は一気に増えます**が、多くの場合 **ロジックは変わらず、見た目（空白・改行・クォートなど）だけ**が変わります。

差分が膨らみやすい主な要因は次のとおりです。

| 要因 | 差分が大きくなりやすい理由 |
| ---- | -------------------------- |
| 初回の一括整形 | 対象ファイルが一度にすべて変わる |
| 改行コードの統一（`.prettierrc` の `endOfLine: "lf"` など） | Windows で CRLF だったファイルがまとめて変わる |
| 除外設定の不足 | 生成物やベンダーまで整形対象になる |
| 機能変更との混在 | 整形と実装が同じ PR に入るとレビューが困難になる |

`.prettierignore` で `vendor`・`*.min.js`・PHP などを除外しているのと同様に、既存プロジェクトでも**対象外を先に決める**ことが重要です。

### 既存プロジェクトへの導入の進め方（おすすめ）

| 段階 | やること |
| ---- | -------- |
| 1. 導入前 | `.prettierrc` のルールと `.prettierignore` をチームで合意する |
| 2. 初回 | 整形だけの 1 コミット（または 1 PR）にする。機能追加・バグ修正と**混ぜない** |
| 3. 以降 | 保存時の自動整形（`.vscode/settings.json`）で、触ったファイルから揃えていく |

#### このサンプルについて（拡張機能のみ）

このリポジトリは **Prettier の VS Code / Cursor 拡張機能** と設定ファイルだけで動かす想定です。  
`package.json` や Node.js は**入れていません**。

そのため、よく紹介される次のような **CLI（コマンドライン）での一括整形** は、そのままでは使えません。

```bash
npx prettier --write .   # ← Node.js と prettier パッケージが必要
```

`npx` は Node.js が入っている環境で、npm 経由で Prettier を取りにいく仕組みです。拡張機能だけの構成では動きません。

#### 既存プロジェクトを「拡張機能だけ」で進める場合

一括で全ファイルを一度に整えるのは難しいので、次のような進め方が現実的です。

| やり方 | 内容 |
| ------ | ---- |
| **触ったファイルから順次** | 設定を置いたあと、編集して保存するたびにそのファイルだけ整形される。差分は少しずつ増える |
| **フォルダ内を手動で** | 対象ファイルを開き、整形（Mac: `⇧⌥F` / Windows: `Shift+Alt+F`）または保存を繰り返す。ファイル数が少なければこれでも足りる |
| **初回だけ CLI を使う** | 一括適用したいタイミングだけ、プロジェクトに `prettier` を dev 依存で入れて `npx prettier --write .` を実行する（後述） |

「初日にリポジトリ全体を 1 コミットで揃える」は CLI 向けの話が多く、**拡張機能のみの運用では必須ではありません**。  
小さなプロジェクトや、これから触るファイルが限られている現場なら、**保存時整形に任せる**だけでも十分なことが多いです。

#### 参考: 既存プロジェクトで初回だけ一括整形したい場合（Node.js あり）

チームで「最初に一度だけ全部揃えたい」ときは、プロジェクトのルートで Prettier を入れてから実行します（以降の日常作業は拡張機能のままでも構いません）。

```bash
npm init -y
npm install -D prettier
npx prettier --write .
git commit -m "chore: プロジェクト全体に Prettier を適用"
```

PR の説明には「ロジックの変更なし・Prettier の一括適用」と書いておくと、レビューしやすくなります。

**避けたいこと:** 既存リポジトリの初日に、機能開発と Prettier の一括整形を同じ PR に入れること。

### まとめ

- **試す場所** → 新規・小さなプロジェクト（このサンプル）から始めるのがよい
- **このサンプルの運用** → 拡張機能 + 保存時整形。CLI の一括整形は不要
- **本番の既存プロジェクト** → 導入してよい。一括は Node + CLI が使えるときの選択肢。拡張機能だけなら**触ったファイルから順次**でもよい
- **SCSS で CSS を生成している場合** → 編集・整形の対象は SCSS 側に寄せ、生成された CSS は直接いじらない（下記参照）

## スタイル（CSS）を SCSS で書いている場合

このサンプルでは、スタイルは `common/sass/sample.scss` を編集します。  
`common/css/sample.css` は自動で作られるファイルなので、**こちらは直接いじらない**でください。

SCSS を CSS に変換する機能を使う場合は、別途 [Live Sass Compile](https://marketplace.visualstudio.com/items?itemName=ritwickdey.live-sass) という拡張機能も入れてください（設定はすでに `.vscode/settings.json` に入っています）。

## 保存しても整わないとき

- Prettier の拡張機能が入っているか、もう一度確認する
- プロジェクトのフォルダごと開いているか確認する（ファイル 1 つだけ開いていると設定が効かないことがあります）
- スタイルを直したいのに `sample.css` を開いていないか確認する（編集するのは `sample.scss` 側です）
- **PHP ファイル**を試したが整わない → Prettier は PHP 非対応です（上記「PHP について」を参照）

それでも直らない場合は、分かりそうな人にご相談ください。
